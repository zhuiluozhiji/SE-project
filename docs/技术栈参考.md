# 技术栈参考

推荐你们这个项目统一用：
> **前端：Vue 3 + Vite + Element Plus**
> **后端：Python + FastAPI**
> **爬虫/OCR/推荐：Python**
> **数据库：MySQL**
> **接口文档：FastAPI 自带 Swagger/OpenAPI**

原因很简单：你们项目涉及**爬虫、OCR、推荐规则、日历接口、活动管理**，Python 生态更顺手；FastAPI 会自动生成 OpenAPI/Swagger 文档，方便前后端联调；Vue 官方也推荐用 `create-vue` 快速创建基于 Vite 的项目，Element Plus 本身就是 Vue 3 的组件库，适合快速做管理后台、表格、表单、筛选器和日历页面。([FastAPI](https://fastapi.tiangolo.com/?utm_source=chatgpt.com))

## 二、前端推荐
### 推荐语言
前端建议用：
```
JavaScript 或 TypeScript
```
如果你们前端同学基础一般，用 **JavaScript** 就够了。
如果前端同学比较熟，建议用 **TypeScript**，后期接口字段更不容易写错。
### 推荐框架
```
Vue 3 + Vite + Element Plus
```
### 为什么推荐 Vue 3？
Vue 对课程项目比较友好，写页面快，组件组织清楚。Vue 官方快速开始文档里也使用 `create-vue` 作为脚手架，并基于 Vite 创建项目。([Vue.js](https://vuejs.org/guide/quick-start?utm_source=chatgpt.com))
### 为什么推荐 Element Plus？
你们项目需要大量现成组件：

- 表格；

- 表单；

- 日期选择器；

- 弹窗；

- 标签；

- 按钮；

- 分页；

- 后台管理页面；

- 活动筛选器。


Element Plus 是 Vue 3 组件库，可以大幅减少自己写 UI 组件的时间。([Element Plus](https://element-plus.org/?utm_source=chatgpt.com))
### 前端具体工具
| 工具 | 作用 |
| --- | --- |
| Vite | 前端项目构建和启动工具 |
| Vue Router | 页面路由，例如首页、活动详情页、日历页 |
| Pinia | 前端状态管理，例如保存登录用户、token、筛选条件 |
| Axios | 向后端发送 HTTP 请求 |
| Element Plus | UI 组件库 |
| FullCalendar / Element Calendar | 日历展示 |
| ECharts | 后台统计图表、个人学术轨迹图表 |

Pinia 是 Vue 生态里常用的状态管理工具，官方文档强调它有 TypeScript 支持、Devtools 支持和模块化设计；Axios 是浏览器和 Node.js 都可用的 HTTP Client，适合前端调用你们的 `/api/v1/...` 接口。([Pinia](https://pinia.vuejs.org/?utm_source=chatgpt.com))

## 三、后端推荐
### 推荐语言
```
Python
```
### 推荐框架
```
FastAPI
```
### 为什么推荐 FastAPI？
你们这个项目后端要做很多 API，例如：
```
GET /api/v1/activities
GET /api/v1/activities/{id}
POST /api/v1/schedules/add-activity
POST /api/v1/courses/ocr
GET /api/v1/recommendations/activities
```
FastAPI 很适合这种前后端分离项目。它可以自动生成 OpenAPI 文档，并提供 Swagger UI 和 ReDoc 这类交互式接口文档页面，前端同学可以直接在浏览器里查看和测试接口。([FastAPI](https://fastapi.tiangolo.com/?utm_source=chatgpt.com))
### 后端具体工具
| 工具 | 作用 |
| --- | --- |
| FastAPI | 后端 Web 框架，写 API |
| Uvicorn | 启动 FastAPI 服务 |
| Pydantic | 定义请求和响应的数据结构 |
| SQLAlchemy | ORM，用 Python 操作 MySQL |
| Alembic | 数据库迁移，管理表结构变化 |
| PyJWT / python-jose | 生成和校验登录 token |
| passlib / bcrypt | 密码加密 |
| python-multipart | 支持文件上传，例如课表截图 |
| ics.py / icalendar | 生成 ICS 日历文件 |

SQLAlchemy 是 Python 的 SQL 工具包和 ORM；Alembic 是和 SQLAlchemy 配套使用的轻量级数据库迁移工具，适合管理你们后续新增字段、改表结构这类操作。([SQLAlchemy](https://www.sqlalchemy.org/?utm_source=chatgpt.com))

## 四、爬虫推荐
### 推荐语言
```
Python
```
### 推荐工具
第一阶段建议简单一点：
```
requests + BeautifulSoup
```
如果后面网页结构复杂，再考虑：
```
Scrapy
```
### 为什么不用一开始就上 Scrapy？
因为你们第一阶段只爬一个学院，比如计院官网。
目标不是做一个很复杂的分布式爬虫，而是先证明：
```
计院官网页面 → 爬虫抓取 → 清洗字段 → 存入 activity 表 → 前端展示
```
所以第一版用 `requests + BeautifulSoup` 就够了。
### 爬虫工具分工
| 工具 | 作用 |
| --- | --- |
| requests | 请求网页 HTML |
| BeautifulSoup | 解析网页内容 |
| re | 正则提取时间、地点、主讲人 |
| pandas | 临时清洗数据、导出 CSV |
| SQLAlchemy / pymysql | 把爬到的数据写入 MySQL |
| schedule / APScheduler | 后期定时运行爬虫，可选 |

## 五、OCR 推荐
### 推荐语言
```
Python
```
### 推荐工具
```
PaddleOCR
```
你们项目里有“课表截图识别”的需求，PaddleOCR 比较适合中文 OCR 场景。PaddleOCR 官方说明它是基于 PaddlePaddle 的多语言 OCR 工具，并支持多语言识别；其 GitHub 页面也说明它可以把图片/PDF转换成结构化数据，适合你们这种“课表截图 → 课程信息”的场景。([PaddlePaddle](https://paddlepaddle.github.io/PaddleOCR/main/en/index.html?utm_source=chatgpt.com))
不过我建议你们这样安排优先级：
```
第一步：支持手动添加课程
第二步：支持 CSV / Excel 导入课表
第三步：支持课表截图 OCR
```
OCR 不要作为第一周核心任务，否则很容易卡住主流程。

## 六、数据库推荐
### 推荐数据库
```
MySQL
```
原因：

- 大家课程里通常接触过；

- 和 Java / Python 都好连接；

- 适合用户、活动、课表、日程这种结构化数据；

- 课程项目验收比较稳。

### 数据库配套工具
| 工具 | 作用 |
| --- | --- |
| MySQL | 主数据库 |
| Navicat / DBeaver | 可视化查看数据库 |
| SQLAlchemy | 后端 ORM |
| Alembic | 数据库迁移 |
| schema.sql | 建表脚本 |
| seed.sql | 初始化测试数据 |

如果你们不熟 Alembic，第一阶段可以先不用，直接维护：
```
database/schema.sql
database/seed.sql
```
这样更容易上手。

## 十、不太建议的方案
### 不建议一：前端 React，后端 Java，爬虫 Python，OCR Python
不是不能做，但你们组会变成三套技术栈：
```
React / Java / Python
```
沟通成本会高。
### 不建议二：后端用 Django
Django 也很好，但对你们这个项目来说有点重。你们主要是写 API，不是做传统模板渲染网站，FastAPI 更轻。
### 不建议三：一开始就做复杂推荐模型
第一阶段不要搞机器学习训练。
推荐直接用规则：
```
兴趣标签匹配 + 热度 + 时间临近 + 校区 + 冲突惩罚
```
这样更可控，也更容易答辩解释。
