# 开发计划书总体写作原则

这份文档应围绕下面几个问题展开：

```text
系统准备采用什么架构？
前端、后端、爬虫、数据库分别怎么开发？
每个模块由谁负责？
数据库和接口先做哪些？
前后端字段如何统一？
代码如何协作？
```

不要重点写：

```text
用户有哪些需求
系统有哪些用例
系统有哪些非功能性需求
验收标准是什么
风险分析
项目交付物
详细测试计划
详细进度计划
```

这些内容要么需求分析报告已经写过，要么不在你们现在保留的目录中。

# 2 项目开发目标与阶段范围

这一章要写清楚：你们第一阶段到底准备开发哪些内容，哪些内容暂时不做。

## 2.1 开发目标

这一节从开发角度写目标，而不是从用户需求角度写目标。

建议写：

本阶段开发目标是在课程项目周期内实现一个可运行、可展示、可联调的校园学术活动智能推荐平台原型。系统应优先跑通“活动数据入库—活动浏览检索—活动详情查看—推荐展示—课表导入—冲突检测—加入日程—日历展示—ICS 导出—后台管理”的核心开发链路。

注意这里写的是“开发链路”，不是“用户需求列表”。

## 2.3 后续迭代范围

这一节写哪些功能暂时不作为第一阶段主任务。

建议写：

以下功能作为后续迭代方向，第一阶段只保留入口或预留接口：

1. 统一身份认证 SSO/CAS；
2. 地图 API 精确通勤测算；
3. 学术组队；
4. 笔记共享与活动评价；
5. 个人学术轨迹；

## 2.4 开发优先级原则

这一节写你们开发时如何取舍。

建议写：

本项目采用“主流程优先、展示效果优先、接口稳定优先”的开发原则。开发顺序上，优先保证活动列表、活动详情、课表导入、冲突检测、日历展示和 ICS 导出等核心链路可运行；对于 OCR、复杂推荐、跨校区通勤、社交互动等功能，采用预留接口或简化实现方式，避免影响主流程完成。

# 3 系统开发架构

这一章是最接近模板的部分。模板中有体系结构风格、层级结构、服务器端和客户端划分，你们可以借鉴这种写法。

## 3.1 总体架构

写系统采用什么架构。

建议写：

本系统采用前后端分离的 Web 应用架构，整体划分为前端客户端、后端服务端、数据采集子系统、数据库存储层和外部依赖接口层。前端负责页面展示和用户交互，后端负责业务逻辑处理和接口服务，数据采集子系统负责从学院网站抓取活动信息，数据库负责存储用户、活动、课表、日程、标签等结构化数据。

可以配一张图：

```text
用户浏览器
   ↓
前端 Vue/React 页面
   ↓ HTTP API
后端 FastAPI/Spring Boot 服务
   ↓
MySQL 数据库
   ↑
爬虫数据导入模块
```

## 3.2 前端子系统

写前端负责什么页面、什么交互。

建议内容：

前端子系统负责用户可见界面和交互逻辑，主要包括登录注册页面、活动列表页、活动详情页、日历页面、课表导入页面、个人中心页面和简化后台管理页面。前端通过统一封装的 API 请求模块与后端通信，并根据接口返回的活动状态、冲突状态和推荐状态展示不同颜色和标签。

可以列前端页面：

| 页面    | 主要功能                       |
| ----- | -------------------------- |
| 登录注册页 | 用户登录、注册、Token 保存           |
| 首页    | 推荐活动、热门活动展示                |
| 活动列表页 | 搜索、筛选、排序、分页                |
| 活动详情页 | 查看活动详情、一键加入日程              |
| 日历页   | 展示课程、活动、冲突状态               |
| 课表导入页 | 手动录入、CSV/Excel 上传、OCR 上传入口 |
| 后台页面  | 活动新增、编辑、下架、爬虫数据修正          |

## 3.3 后端子系统

写后端服务怎么分层。

建议写：

后端子系统采用分层结构，分为接口层、业务逻辑层、数据访问层和工具层。接口层负责接收 HTTP 请求并返回统一格式响应；业务逻辑层负责用户认证、活动管理、推荐计算、冲突检测、课表导入和 ICS 导出；数据访问层负责与数据库交互；工具层封装时间处理、文件解析、Token 生成、ICS 文件生成等通用能力。

可以列后端模块：

| 模块              | 主要职责             |
| --------------- | ---------------- |
| auth            | 登录、注册、Token 鉴权   |
| users           | 当前用户信息、用户兴趣      |
| activities      | 活动列表、详情、CRUD     |
| recommendations | 规则推荐、推荐理由        |
| schedules       | 日程查询、加入日程、冲突检测   |
| courses         | 课表导入、课程保存        |
| crawler         | 爬虫触发、爬虫记录        |
| admin           | 后台管理接口           |
| utils           | 时间处理、ICS 生成、文件解析 |

## 3.4 数据采集子系统

写爬虫怎么开发。

建议写：

数据采集子系统第一阶段以单一学院官网为目标，优先实现“抓取—解析—清洗—去重—入库”的基本流程。爬虫模块需提取活动标题、发布时间、活动时间、地点、主讲人、来源链接、学院、类别等字段，并将结果写入活动表。对于字段缺失或格式不统一的情况，允许后台人工修正。

这里不要写复杂分布式爬虫，第一阶段只写“单学院可运行爬虫”。

## 3.5 数据存储与接口层

写数据库和接口的整体关系。

建议写：

数据存储层以 MySQL 为主，围绕用户、活动、课表、日程、标签、报名记录和爬虫记录设计核心数据表。后端通过 ORM 或数据库访问层统一操作数据库，避免业务代码直接拼接 SQL。接口层统一采用 `/api/v1` 前缀，并使用统一返回格式，保证前后端联调时字段稳定。

## 3.6 外部依赖接入策略

写外部依赖如何处理。

建议写：

本项目涉及的外部依赖包括学院官网数据源、OCR 工具、日历文件格式和后续可能接入的统一身份认证、地图 API 等。第一阶段对外部依赖采用“可替换、可降级”的接入策略：课表导入优先支持 CSV/Excel，OCR 作为增强功能；统一身份认证暂不强依赖，先使用本地账号系统；地图 API 暂不作为核心功能，只预留字段和接口扩展点。

# 4 模块开发计划

这一章写每个模块怎么开发。每个模块建议统一用这个格式：

```text
模块目标：
主要开发内容：
关键接口或数据：
实现重点：
与其他模块的关系：
```

## 4.1 用户认证与权限模块

写什么：

* 简化登录注册；
* 密码加密存储；
* Token 生成；
* 获取当前用户；
* 区分普通用户和管理员。

建议写：

该模块负责用户身份认证和基础权限判断。第一阶段采用账号密码登录方式，登录成功后后端返回 Token，前端保存 Token 并在后续请求中携带。管理员接口需要进行角色校验，普通用户不能访问活动管理、爬虫触发等后台接口。

## 4.2 活动信息管理模块

写什么：

* 活动列表；
* 活动详情；
* 新增活动；
* 编辑活动；
* 下架活动；
* 爬虫活动人工修正。

建议写：

该模块是系统的数据展示基础，负责对 activity 表中的活动信息进行增删改查。前台用户通过活动列表和详情页查看活动，管理员通过后台页面维护活动字段和状态。爬虫导入的数据也需要进入该模块统一管理。

## 4.3 搜索筛选与排序模块

写什么：

* keyword；
* category；
* campus；
* college；
* tag；
* time range；
* sort_by。

建议写：

该模块负责活动列表的查询能力。后端根据前端传入的查询参数动态构造查询条件，并返回分页结果。排序方式第一阶段支持按活动时间、热度分、推荐分排序。

注意：这里不是写“用户需要搜索活动”，而是写“后端如何提供搜索筛选能力，前端如何调用”。

## 4.4 推荐模块

写什么：

* 规则推荐；
* 推荐分计算；
* 推荐理由；
* 冲突惩罚；
* 不做复杂模型。

建议写：

第一阶段推荐模块采用规则推荐策略，不引入复杂机器学习模型。推荐分由兴趣标签匹配、活动热度、时间临近程度、学院/专业相关性和冲突惩罚等因素组成。接口返回推荐活动列表和简要推荐理由，前端在首页或活动列表中展示。

可以写一个公式：

```text
推荐分 = 标签匹配分 + 热度分 + 时间临近分 + 学院相关分 - 冲突惩罚分
```

## 4.5 课表导入模块

写什么：

* 手动录入；
* CSV/Excel 导入；
* OCR 作为增强；
* 解析后写入 course_schedule。

建议写：

课表导入模块用于将用户课程信息转化为系统可识别的日程数据。第一阶段优先实现手动录入和 CSV/Excel 导入，保证课程名、星期、节次、周次、地点等字段能够落库。OCR 上传作为增强功能，识别结果需要允许用户确认和修改后再保存。

## 4.6 日历与冲突检测模块

写什么：

* 日历事件统一模型；
* 课程、活动、个人日程；
* 时间冲突判断；
* 冲突列表返回；
* 前端颜色标识。

建议写：

该模块负责管理用户个人日程，并在用户加入活动时检测活动时间是否与已有课程或活动冲突。后端统一返回日历事件结构，前端根据事件类型和冲突状态显示不同颜色。冲突判断采用时间区间重叠规则：

```text
new_start < old_end AND new_end > old_start
```

这个规则已经在你们接口规范草案中出现，可以直接作为开发实现依据。

## 4.7 ICS 导出模块

写什么：

* 单个活动导出；
* 用户完整日程导出；
* 生成 `.ics` 文件；
* 设置时间、地点、标题、描述。

建议写：

ICS 导出模块负责将系统中的活动或用户日程转换为标准日历文件。第一阶段支持导出用户当前日程列表，生成可被第三方日历软件导入的 `.ics` 文件。后端根据 schedule_event 数据构造日历事件，前端通过下载链接获取文件。

## 4.8 后台管理模块

写什么：

* 管理员查看活动；
* 新增、编辑、下架；
* 爬虫数据修正；
* 简单统计。

建议写：

后台管理模块面向管理员和活动维护人员，第一阶段提供简化后台页面，用于维护活动数据、修正爬虫导入字段、设置活动状态和查看基础统计信息。后台接口需要进行管理员权限校验。

## 4.9 爬虫与数据清洗模块

写什么：

* 选取一个学院；
* 抓取字段；
* 清洗；
* 去重；
* 入库；
* 记录 crawl_record。

建议写：

爬虫模块第一阶段选择计算机学院官网或活动通知页面作为数据源，抓取活动标题、时间、地点、主讲人、来源链接和发布时间等字段。清洗后写入 activity 表，并记录爬虫运行状态。对于重复活动，可根据标题、时间和来源链接进行简单去重。

# 5 技术路线与开发环境

这一章写你们准备用什么技术，不要写太泛。

## 5.1 前端技术路线

写：

* Vue 3 + Vite + Element Plus，或 React + Vite + Ant Design；
* 路由；
* 状态管理；
* API 封装；
* 日历组件选择。

示例：

前端计划采用 Vue 3 + Vite + Element Plus。Vue 3 用于组件化页面开发，Vite 用于快速构建，Element Plus 提供表单、表格、日期选择器、弹窗等基础组件。日历页面可优先采用 Element Plus Calendar 或 FullCalendar，若时间紧张则实现简化月视图。

## 5.2 后端技术路线

写：

* FastAPI 或 Spring Boot；
* RESTful API；
* Swagger/OpenAPI；
* ORM；

# 8 版本控制与协作规范

这一章写 Git 规范。

## 8.1 Git 仓库结构

建议写：

项目采用前后端分离目录结构：

```text
campus-academic-recommender/
├── frontend/
├── backend/
├── crawler/
├── database/
├── docs/
├── tests/
├── README.md
└── docker-compose.yml
```

这个目录结构与你们已有开发分工草案保持一致。

## 8.2 分支命名规范

建议写：

```text
main
frontend/calendar-view
frontend/activity-list
backend/auth-api
backend/activity-api
backend/schedule-conflict
crawler/cs-zju
docs/interface-spec
fix/login-bug
```

原则：

* 不直接在 main 上开发；
* 每个功能一个分支；
* 分支名体现模块和功能。

## 8.3 提交信息规范

建议写：

```text
feat: 新增功能
fix: 修复问题
docs: 修改文档
style: 样式调整
refactor: 代码重构
test: 测试相关
chore: 配置或杂项
```

示例：

```text
feat: add activity list api
feat: implement schedule conflict detection
fix: fix calendar event rendering
docs: update interface specification
```

## 8.4 合并规则

建议写：

合并到主分支前，开发者需要确保本地可以正常运行，接口没有破坏已有功能。涉及接口和数据库字段变化的提交，必须同步更新接口文档和数据库说明。合并时优先通过 Pull Request 或 Merge Request 进行，避免直接推送主分支。

## 8.5 代码冲突处理

建议写：

当多人修改同一文件产生冲突时，由该模块负责人协调处理。对于接口文件、数据库脚本和公共工具类等高频冲突文件，修改前应先同步当前分支并告知相关成员。冲突解决后需要重新运行相关功能，避免合并后出现隐藏错误。
