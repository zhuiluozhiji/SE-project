## 4 功能模块开发计划

这一章要从“开发角度”写每个模块，而不是只写需求。

建议每个模块都用统一格式：

```text
模块目标：
主要功能：
输入输出：
负责人：
优先级：
第一阶段交付标准：
```

例如：

### 4.1 活动浏览与检索模块

模块目标：
为用户提供统一的校园学术活动浏览入口，支持用户通过关键词、时间、地点、学院、类别、标签等条件筛选活动。

主要功能：

1. 活动卡片展示；
2. 活动详情展示；
3. 关键词搜索；
4. 按学院、类别、校区、时间筛选；
5. 按热度、时间、推荐分排序；
6. 分页加载。

第一阶段交付标准：

1. 能从数据库读取活动数据；
2. 前端能展示活动列表；
3. 搜索和筛选条件能正常生效；
4. 活动详情页能展示完整字段。

### 4.2 日历与冲突检测模块

模块目标：
将用户课程、已加入活动和推荐活动统一展示在日历中，并在用户加入活动时判断时间冲突。

主要功能：

1. 个人日程查询；
2. 课程日程展示；
3. 活动加入日程；
4. 时间冲突检测；
5. 冲突活动红色标识；
6. ICS 文件导出。

冲突检测逻辑可以直接写：

```text
若 new_start < old_end 且 new_end > old_start，则两个时间段存在冲突。
```

这与你们接口规范草案中的冲突检测规则一致。

## 10 风险分析与应对措施

这一章建议认真写，因为老师会觉得你们考虑得比较工程化。

| 风险          | 表现            | 应对措施                         |
| ----------- | ------------- | ---------------------------- |
| OCR 识别效果不稳定 | 课表截图格式复杂，识别失败 | 第一阶段优先支持 CSV/Excel，OCR 作为加分项 |
| 爬虫数据格式不统一   | 学院官网字段混乱      | 先只做计院官网，必要时使用人工样例数据兜底        |
| 前后端接口变化频繁   | 联调成本高         | 每周设置接口冻结点，接口变动必须同步文档         |
| 推荐算法过复杂     | 开发时间不足        | 第一阶段采用规则推荐，不做复杂机器学习          |
| 日历组件联调困难    | 事件渲染和冲突标识复杂   | 先实现简化月视图或周视图                 |
| 进度延期        | 功能过多导致无法收尾    | 严格区分 MVP 功能和第二阶段功能           |

# 四、你们这份文档最应该突出的主线

整份开发计划书不要写散，应该始终围绕这一条主线：

```text
信息聚合 → 活动检索 → 个性化推荐 → 课表导入 → 冲突检测 → 日程管理 → ICS 导出
```

也就是：

用户先能看到活动，
然后能找到自己感兴趣的活动，
再判断自己有没有时间参加，
最后把活动加入自己的日历。

这条主线是你们项目区别于普通“活动发布系统”的核心价值。

# 六、最推荐的写法策略

你们可以把这份项目开发计划书写成三种材料的融合：

| 来源          | 在开发计划书中承担的作用                         |
| ----------- | ------------------------------------ |
| 需求分析报告      | 提供项目背景、用户需求、功能边界、验收依据                |
| 开发分工与接口规范草案 | 提供 MVP 范围、人员分工、接口、数据库、进度计划           |
| 网上书店模板      | 提供报告组织形式，尤其是体系结构、服务器设计、数据库设计、UI 设计写法 |

这样写出来会比较完整，也比较符合软件工程课程作业的要求。
