# 4 服务器端设计

这一章对应模板第 2 章。模板中的服务器端设计包含服务器架构、服务器处理流程和数据库设计。你们要在这个基础上加入接口设计，因为你们是前后端分离项目。

## 4.1 服务器端总体架构

写后端服务整体如何组织。

建议写：

服务器端采用分层架构，主要包括表现层接口、业务逻辑层、数据访问层和工具支撑层。表现层接口负责接收前端请求并返回统一格式响应；业务逻辑层负责用户认证、活动管理、推荐计算、课表导入、冲突检测、日程管理和 ICS 导出；数据访问层负责与数据库交互；工具层封装文件解析、时间处理、Token 生成和 ICS 文件生成等通用能力。

可以给一个结构：

```text
Controller / API Router
        ↓
Service
        ↓
Repository / Mapper
        ↓
MySQL
```

## 4.2 表现层接口设计

这里的“表现层”不是前端页面，而是服务器端对外暴露的接口层。

写：

* 接收 HTTP 请求；
* 参数校验；
* Token 校验；
* 调用业务层；
* 返回统一响应。

建议写：

表现层接口负责处理来自前端页面和后台管理页面的 HTTP 请求，包括活动列表查询、活动详情获取、课表导入、冲突检测、加入日程、推荐活动获取和后台活动管理等。接口层不直接处理复杂业务逻辑，而是完成参数解析、权限校验和响应封装，并将具体任务交由业务逻辑层完成。

## 4.3 业务逻辑层设计

写核心业务如何拆。

建议列：

| 业务服务                  | 主要职责                |
| --------------------- | ------------------- |
| AuthService           | 登录、注册、Token 生成、权限判断 |
| ActivityService       | 活动列表、详情、CRUD、状态维护   |
| RecommendationService | 规则推荐、推荐分计算、推荐理由     |
| ScheduleService       | 日程查询、加入活动、冲突检测      |
| CourseService         | 课表导入、课程解析、课程保存      |
| CrawlerService        | 爬虫触发、数据清洗、爬虫记录      |
| AdminService          | 后台管理、活动修正、下架操作      |
| IcsService            | 日历文件生成              |

## 4.4 数据访问层设计

写后端如何访问数据库。

建议写：

数据访问层负责封装数据库操作，避免业务层直接拼接 SQL。每类核心实体设置对应的数据访问对象，例如 UserRepository、ActivityRepository、ScheduleRepository、CourseRepository 等。数据访问层需要支持基础增删改查、分页查询、条件筛选和事务处理。

如果用 Spring Boot，可以写 Mapper/Repository；如果用 FastAPI，可以写 SQLAlchemy Model + CRUD Repository。

## 4.5 数据库设计

### 4.5.1 数据库设计原则

写原则，不要写过细。

建议写：

数据库设计遵循结构清晰、字段统一、减少冗余、便于查询和便于扩展的原则。活动、用户、课表、日程、标签等核心数据分别存储，避免把不同类型的数据混在同一张表中。时间字段统一使用 `start_time`、`end_time` 等命名，便于后端计算冲突和前端展示。

### 4.5.2 核心数据表设计

建议用表格列核心表即可。

| 表名              | 作用                        |
| --------------- | ------------------------- |
| user            | 存储用户账号、角色、学院、专业等信息        |
| activity        | 存储活动标题、描述、时间、地点、主讲人、来源等信息 |
| activity_tag    | 存储活动与标签的关系                |
| user_interest   | 存储用户兴趣标签                  |
| course_schedule | 存储用户导入的课程表                |
| schedule_event  | 存储用户日程事件，包括课程、活动、个人日程     |
| registration    | 存储用户加入或报名活动的记录            |
| crawl_record    | 存储爬虫运行记录                  |
| admin_log       | 存储管理员操作日志                 |

这些表与你们已有接口规范草案中的数据库核心表保持一致。

### 4.5.3 数据库开发顺序

写开发先后顺序。

建议写：

数据库开发优先围绕主流程展开：

1. 建立 `user` 表，支持登录和角色判断；
2. 建立 `activity` 表，支持活动列表和详情；
3. 建立 `activity_tag` 与 `user_interest`，支持标签筛选和推荐；
4. 建立 `course_schedule`，支持课表导入；
5. 建立 `schedule_event`，支持日历展示和加入日程；
6. 建立 `registration`，记录用户活动加入关系；
7. 建立 `crawl_record` 和 `admin_log`，支持爬虫和后台管理。

```

# 9 版本控制与协作规范

campus-academic-recommender/

```

这个目录结构与你们已有开发草案中的项目目录建议一致。

## 9.2 分支命名规范

建议写：

```text
main
frontend/calendar-view
frontend/activity-list
backend/auth-api
backend/activity-api
backend/schedule-conflict
crawler/cs-zju
docs/interface-spec
fix/login-bug
```

原则：

* 不直接在 `main` 上开发；
* 每个功能单独开分支；
* 分支名体现模块和功能。

## 9.3 提交信息规范

写：

```text
feat: 新增功能
fix: 修复问题
docs: 修改文档
style: 样式调整
refactor: 代码重构
test: 测试相关
chore: 配置或杂项
```

示例：

```text
feat: add activity list api
feat: implement schedule conflict detection
fix: fix calendar event rendering
docs: update interface specification
```

## 9.4 合并规则

写：

合并到主分支前，开发者需要保证本地可以正常运行，且没有破坏已有接口。涉及接口字段或数据库字段变化的提交，必须同步更新接口文档和数据库说明。合并时优先通过 Pull Request 或 Merge Request 进行，避免直接推送主分支。

## 9.5 代码冲突处理

写：

当多人修改同一文件导致冲突时，由对应模块负责人协调解决。对于接口文件、数据库脚本、公共工具类等容易产生冲突的文件，修改前应先同步最新代码并告知相关成员。冲突解决后需要重新运行相关功能，避免合并后引入隐藏错误。
